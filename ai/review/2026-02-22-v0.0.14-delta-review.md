## New Issues in This Diff

### Correctness / Safety

**[ERROR] src/commands/logs.rs:85-91 — `pager_args` borrows from `pager_env` which is dropped too soon (dangling lifetime)**

`pager_env` is a `String` owned in the function. `parts` is a `split_whitespace()` iterator over `&str` slices into `pager_env`. `pager_args: Vec<&str>` collects those slices. Then `pager_env` remains alive until end of scope, so the borrow is valid — but if `pager_cmd == "less"`, the literal `"-R"` pushed into `pager_args` is a `&'static str`, which is fine. The actual lifetime issue here is subtler: all of `pager_cmd`, `pager_args`, and `pager_env` must be in scope when `Command::new(pager_cmd).args(&pager_args)` is called.

On re-inspection this compiles and is safe — the lifetimes work out because `pager_env` lives until end of the function. No actual bug, but the `Vec<&str>` containing lifetime-tied slices mixed with a pushed `&'static str` is an unusual pattern. Confidence drops below threshold; noting for awareness only.

**[WARN] src/commands/logs.rs:90 — `-R` check compares `&str` against `pager_args: Vec<&str>` after split**

If `PAGER="less --RAW-CONTROL-CHARS"` (long form of `-R`), the check `!pager_args.contains(&"-R")` is false so `-R` is not appended — correct. But if `PAGER="less -r"` (lowercase, renders only ANSI), `-R` would still be appended, giving `less -r -R` which is redundant but harmless. This is a minor edge case, not a bug.

**[ERROR] src/daemon/spawner.rs:80-88 — Spawn failure during `run_job` can still race with `stop_job`**

```rust
tokio::spawn(async move {
    if let Err(e) = run_job(&state_clone, job_id.clone(), command, cwd, timeout_secs).await {
        error!("Job {} failed: {}", job_id, e);
        let db = state_clone.db.lock().unwrap();
        if let Err(db_err) = db.update_finished(&job_id, Status::Failed, None) {
            error!("Failed to mark job {} as failed: {}", job_id, db_err);
        }
    }
});
```

`run_job` fails early (e.g., bad `cwd`, can't open log file) before inserting into `running_jobs`. Meanwhile a `jb stop <id>` arrives: `stop_job` checks `running_jobs`, finds nothing, returns `Response::Error("Job is not running")`. The stop command falls through to its direct-kill fallback in `commands/stop.rs:50-54`, reads `job.status == Pending` (it hasn't transitioned yet), and calls `db.update_status(&job.id, Status::Stopped)`. Then `run_job`'s error handler fires and calls `update_finished(Failed)`, overwriting Stopped with Failed. The job ends up Failed even though the user requested it be Stopped.

This is an edge case (tiny window, requires `jb stop` on a job that fails to start) and is benign (wrong terminal status, not a crash). Confidence: ~70%. Noting but not marking ERROR — downgraded to WARN.

**[WARN] src/daemon/spawner.rs:80-88 — Race between `run_job` error handler and `stop_job` (see above)**

Fix: before calling `update_finished(Failed)` in the error handler, check that the current status is still Pending/Running (not already a terminal state set by stop):

```rust
if let Ok(Some(job)) = db.get(&job_id) {
    if !job.status.is_terminal() {
        let _ = db.update_finished(&job_id, Status::Failed, None);
    }
}
```

**[WARN] src/core/db.rs:183-191 — Status filter branch uses tautological `AND status NOT IN` guard**

```rust
"DELETE FROM jobs WHERE created_at < ?1 AND status = ?2 AND status NOT IN ('running', 'pending')"
```

When `status = ?2` equals `'running'` or `'pending'`, the `AND status NOT IN (...)` makes the whole WHERE clause unsatisfiable — zero rows deleted. This is correct behavior (prevents deleting running jobs), but it works by accident: if a caller passes `Status::Running`, the guard silently prevents deletion rather than rejecting the input. A user running `jb clean --status running` gets "Removed 0 jobs" with no error, which is confusing.

The fix was meant to prevent data corruption, and it does — but it should reject invalid status values at the CLI layer rather than silently no-op at the SQL layer. This is not a correctness bug (the guard works), but the UX is wrong.

Fix: in `commands/clean.rs`, validate that the status filter is a terminal status before proceeding:

```rust
if let Some(ref s) = status_filter {
    if !s.is_terminal() {
        anyhow::bail!(UserError::new(format!(
            "Cannot clean jobs with status '{s}' — only terminal statuses allowed"
        )));
    }
}
```

### Quality / Refactoring

**[WARN] src/commands/logs.rs:218-316 — `follow_logs` still uses `std::thread::sleep` in async context**

This is the outstanding item from P2-12 in the prior review. Still unaddressed. `follow_logs` is called from the synchronous `execute()` which is called from an async `run()` inside `#[tokio::main]`. The blocking `std::thread::sleep(Duration::from_millis(100))` on lines 238 and 315 stalls the Tokio thread. For the CLI this is benign (logs is exclusive, single-task), but it is the one remaining structural issue from the prior review.

**[WARN] src/commands/run.rs:22-25 — `canonicalize()` on non-existent `--dir` panics with opaque OS error**

```rust
let cwd = match dir {
    Some(d) => PathBuf::from(d).canonicalize()?,
    None => env::current_dir()?,
};
```

`canonicalize()` returns `Err` if the path does not exist. The error message from the OS is something like `No such file or directory (os error 2)` with no mention of which path failed. For an agent-facing CLI, this is poor UX.

Fix: wrap with context:

```rust
Some(d) => {
    let p = PathBuf::from(&d);
    p.canonicalize()
        .with_context(|| format!("directory not found: {d}"))?
}
```

**[NIT] src/commands/stop.rs:41 — Error string matching to detect "not running" response is still fragile**

```rust
Response::Error(e) => {
    if !e.contains("not running") {
        anyhow::bail!("{e}");
    }
}
```

This was not addressed in the diff. The daemon returns `Response::Error("Job {id} is not running")` from `server.rs:185`. The stop command inspects this string to decide whether to fall through to the direct-kill path. If the error message changes (e.g., to "is not currently running"), this silently becomes an unhandled error. A `Response::UserError` or a dedicated `Response::NotRunning` variant would be cleaner.

This pattern pre-dates this diff and was noted as P3-17 in the prior review (not addressed there either). Noting it here as it was not cleaned up when `UserError` was added.
